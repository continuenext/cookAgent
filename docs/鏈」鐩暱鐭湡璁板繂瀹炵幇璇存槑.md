# GustoBot 本项目长短期记忆实现说明

## 1. 总览

本项目的“记忆”不是单一组件，而是分层实现：

- 短期记忆（会话级）
  - 目标：保证同一会话的多轮连续对话
  - 载体：LangGraph Checkpointer + Redis
  - 辅助：上下文压缩模块（对话摘要与窗口预算）

- 长期记忆（知识级）
  - 目标：跨会话可复用知识与结构化事实
  - 载体：
    - Milvus（向量语义记忆）
    - Neo4j（图谱关系记忆）
    - MySQL（结构化业务事实）

对应关键实现文件：
- 短期记忆
  - gustobot/application/services/chat_service.py
  - gustobot/application/services/session_service.py
  - gustobot/application/agents/lg_builder.py
  - gustobot/infrastructure/core/context_manager.py
- 长期记忆
  - gustobot/application/services/knowledge_service.py
  - gustobot/application/agents/rag_sub_graph/components/graph_rag/rag_modules/milvus_index_construction.py
  - gustobot/infrastructure/knowledge/recipe_kg/*
  - gustobot/data/init_mysql.sql

## 2. 短期记忆实现（会话上下文）

### 2.1 会话主键与线程绑定

短期记忆的会话主键是 session_id，调用链路中映射为 LangGraph 的 thread_id：

- chat_service.chat
  - configurable = {"thread_id": session_id, ...}
  - 调用 graph.ainvoke(input_state, config=run_config)

这意味着：
- 同一 session_id 的历史会自动累积
- 不同 session_id 的历史天然隔离

### 2.2 LangGraph Checkpointer 持久化

在应用启动时：
- main.py 的 lifespan -> init_graph()
- lg_builder.init_graph() 内创建 AsyncRedisSaver(redis_url=settings.REDIS_URL)
- build_graph(checkpointer=checkpointer) 编译主图

效果：
- 每轮状态（messages/router/answer 等）可回写 Redis
- 服务重启后，只要 Redis 数据仍在，会话上下文可恢复

### 2.3 会话元数据（非消息本体）

session_service.py 中另存了一套会话“元信息”到 Redis：

- Key 前缀
  - gustobot:session:meta:{session_id}
  - gustobot:session:index
  - gustobot:session:user:{user_id}

- 元信息字段
  - session_id
  - title
  - created_at / updated_at
  - message_count
  - user_id
  - status

注意：
- 这套元数据用于会话管理（列表、统计、筛选）
- 消息正文历史仍由 LangGraph Checkpointer 管理

### 2.4 历史读取机制

获取会话历史时：
- session_service.get_session_history(session_id)
- 调用 graph.aget_state({"configurable": {"thread_id": session_id}})
- 从 state.values.messages 反序列化为 user/assistant 列表

兼容：
- HumanMessage / AIMessage 对象
- dict 序列化消息（messages_from_dict）

### 2.5 上下文工程（短期记忆的质量保障）

context_manager.py 提供：
- token 计数
- 历史截断
- 历史摘要
- system/history/retrieval 预算分配
- schema 与检索结果压缩

关键点：
- 历史过长时不会简单丢弃，而是“摘要旧消息 + 保留最近原文”
- 避免多轮后模型遗忘核心约束

## 3. 长期记忆实现（知识与事实）

### 3.1 Milvus：向量语义记忆

职责：
- 存储菜谱文本块 embedding
- 支持相似度检索

典型能力：
- add_documents / similarity_search / delete / stats
- 通过 KB_SIMILARITY_THRESHOLD 做低分过滤

特点：
- 适合语义召回
- 非结构化信息友好
- row_count 通常是“文档块数”，不是“菜谱条数”

### 3.2 Neo4j：图谱关系记忆

职责：
- 存储 Recipe/Ingredient/CookingStep 等实体关系
- 支持多跳图谱查询与 GraphRAG 推理

特点：
- 强关系建模
- 可解释性好
- 适合“关系链路”类问题

### 3.3 MySQL：结构化业务事实记忆

职责：
- recipes / recipe_ingredients / recipe_steps 等结构化事实
- 统计查询、聚合分析、筛选分页

特点：
- 强一致结构化存储
- 易统计、易约束、易回归验证

## 4. 路由与记忆协作方式

主图路由类型（简化）：
- general-query：依赖短期会话上下文
- additional-query：依赖短期上下文 + 追问逻辑
- graphrag-query：短期上下文 + Neo4j + Milvus
- text2sql-query：短期上下文 + MySQL

- image-query / file-query：短期上下文 + 多模态/文件内容

协作原则：
- 短期记忆负责“当前会话连续性”
- 长期记忆负责“跨会话可复用事实”

## 5. 数据生命周期

### 5.1 短期记忆生命周期

1) 用户发送消息
2) session_id 绑定 thread_id
3) graph.ainvoke 执行
4) checkpointer 写 Redis
5) touch_session 更新会话元数据

清理方式：
- 删除会话元信息：session_service.delete_session
- 注意：仅删除 meta 并不等价于彻底删除 checkpoint 历史（需按 checkpointer 键策略额外处理）

### 5.2 长期记忆生命周期

- 图谱/结构化数据
  - 通过初始化脚本、导入脚本、业务写入维护
- 向量库
  - 可增量插入
  - 可 clear 或 rebuild_milvus_index 全量重建

## 6. 一致性与常见误区

### 6.1 常见误区

- 误把 Milvus row_count 当作菜谱总数
  - 实际多为 chunk 数
- 误以为删除 session meta 就删除了所有对话历史
  - 还涉及 checkpointer 存储
- 同一 session_id 反复测试导致“历史干扰”
  - 应使用新 session_id 做 A/B 验证

### 6.2 一致性建议

- 统计类接口优先查 MySQL
- 语义召回优先 Milvus
- 关系推理优先 Neo4j
- 回归测试时明确 session_id 隔离策略

## 7. 配置与调参建议

主要配置见 settings.py：

- 短期记忆相关
  - REDIS_URL
  - GUSTOBOT_MEMORY_TURNS
  - CONTEXT_*（窗口与压缩参数）

- 长期记忆相关
  - MILVUS_*（host/port/collection/index）
  - NEO4J_*（uri/user/password）
  - DATABASE_URL（MySQL）
  - KB_*（top_k/threshold/chunk）

调参建议：
- 对话易遗忘：提高 CONTEXT_SUMMARY_KEEP_RECENT，适度提高 CONTEXT_SUMMARY_THRESHOLD
- 成本过高：降低 CONTEXT_RETRIEVAL_MAX_TOKENS 与 CONTEXT_SCHEMA_MAX_TOKENS
- 无关检索命中：提高 KB_SIMILARITY_THRESHOLD

## 8. 可观测性与排查清单

推荐日志关注点：
- chat_service 开始对话日志（session/user/message）
- lg_builder 路由决策日志（type -> node）
- context_manager 压缩前后 token
- milvus stats 与 search score
- Neo4j/MySQL 查询耗时与返回计数

典型排查路径：

1) “返回像上一轮答案”
- 查 session_id 是否复用
- 查 process_response 是否正确同步 answer

2) “启动后历史丢失”
- 查 Redis 是否重建/清空
- 查 checkpointer 是否正常初始化

3) “知识检索异常”
- 查 Milvus collection/load/index
- 查阈值过滤是否过严
- 必要时重建索引

## 9. 当前实现边界

- 已实现：
  - 会话级短期记忆（LangGraph + Redis）
  - 多后端长期记忆（Milvus/Neo4j/MySQL）
  - 上下文压缩与预算管理

- 待增强（建议）：
  - 会话删除时同步清理 checkpointer 历史键
  - 统一记忆观测面板（短期+长期）
  - 路由结果与检索来源的 debug 可视化

---

## 10. 一句话总结

本项目记忆体系是“短期会话连续性 + 长期知识事实库”双层架构：
- 短期靠 LangGraph Checkpointer in Redis
- 长期靠 Milvus（语义）+ Neo4j（关系）+ MySQL（结构化）
- 通过上下文工程模块在质量、成本、稳定性之间做动态平衡
